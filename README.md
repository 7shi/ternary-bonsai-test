# Ternary-Bonsai-8B Inference Test

A repository for running the 1.58-bit ternary-quantized model [onnx-community/Ternary-Bonsai-8B-ONNX](https://huggingface.co/onnx-community/Ternary-Bonsai-8B-ONNX) locally. Dependency resolution and virtual environment setup are handled automatically via `uv`.

> **English** | [Japanese](README-ja.md)

## Quick Start

The recommended configuration is `model_q2_to_q4 / CPU`. Download the q2 packed ONNX, convert it locally to q4, then run inference.

```bash
uv run python download_q2.py
uv run python convert_q2.py --target-format q4
uv run run_onnx.py --provider cpu --model-source q4
```

## Running Inference

### ONNX Model ([run_onnx.py](run_onnx.py))

| Option | Description |
|--------|-------------|
| `--provider dml` | DirectML (Windows GPU, default) |
| `--provider cpu` | CPU execution provider |
| `--model-source auto` | Auto-select available model (default) |
| `--model-source q4` | q2 → q4 converted model |
| `--model-source q8` | q2 → q8 converted model |
| `--model-source q2fp8` | q2 → dense fp8 converted model |
| `--model-source fp8` | safetensors-derived fp8 export |
| `--model-source fp16` | safetensors-derived fp16 export |
| `--model-source fp32` | safetensors-derived fp32 export |
| `--max-new-tokens N` | Number of tokens to generate (default 10) |
| `--prompt "..."` | Input prompt |

```bash
uv run run_onnx.py --provider cpu --model-source q4 --max-new-tokens 32
```

### Safetensors Model ([run_safetensors.py](run_safetensors.py))

Runs inference directly on the Hugging Face safetensors release `prism-ml/Ternary-Bonsai-8B-unpacked` using CPU.

```bash
uv run run_safetensors.py --max-new-tokens 32
uv run run_safetensors.py --dtype float32 --max-new-tokens 16
```

### q2 Packed ONNX Direct Execution ([run_q2.py](run_q2.py))

A script to confirm that the current ONNX Runtime cannot run the q2 model as-is.

```bash
uv run run_q2.py
```

### Batch Comparison ([compare.bat](compare.bat))

Runs all 13 conditions (safetensors + each ONNX × CPU/DirectML) sequentially.

```bat
compare.bat 10
```

## Model Formats

This repository works with two distinct lineages of models.

### safetensors-derived (Standard ONNX Graph)

Generated from the dense weights of `prism-ml/Ternary-Bonsai-8B-unpacked` using `optimum-cli export onnx`. All operations in the graph are standard ONNX ops.

| Format | Bytes/element | How to generate |
|-------|-------------:|----------------|
| `model_fp32` | 4 | `optimum-cli export onnx --dtype fp32` |
| `model_fp16` | 2 | `optimum-cli export onnx --dtype fp16` |
| `model_fp8` | 1 | Post-process fp16 ONNX with [convert_fp8.py](convert_fp8.py) |

### q2 Packed ONNX-derived (`com.microsoft` Custom Op Graph)

Converted from the 2-bit packed ONNX of `onnx-community/Ternary-Bonsai-8B-ONNX`. The graph retains `com.microsoft` custom ops such as `GroupQueryAttention`, making the graph structure fundamentally different from the safetensors-derived models.

| Format | Conversion script | Format |
|-------|------------------|--------|
| `model_q2_to_q4` | [convert_q2.py](convert_q2.py) `--target-format q4` | `MatMulNBits` q4 |
| `model_q2_to_q8` | [convert_q2.py](convert_q2.py) `--target-format q8` | `MatMulNBits` q8 |
| `model_q2_to_fp8` | [convert_q2_to_fp8.py](convert_q2_to_fp8.py) | Dense fp8 `MatMul` |

`model_q2_to_fp8` expands weights to dense fp8, making the weights themselves equivalent to `model_fp8`, but the attention, normalization, and embedding parts of the graph differ. This structural difference is the root cause of the divergent DirectML behavior ([ONNX.md](ONNX.md)).

## Conversion Steps

### Download q2 Packed ONNX

```bash
uv run python download_q2.py
```

### q2 → q4 / q8 Conversion

```bash
uv run python convert_q2.py --target-format q4
uv run python convert_q2.py --target-format q8
```

### q2 → Dense fp8 Conversion

```bash
uv run python convert_q2_to_fp8.py
```

### safetensors → ONNX Export

```bash
uv run optimum-cli export onnx -m prism-ml/Ternary-Bonsai-8B-unpacked model_fp32 --task text-generation-with-past --pad_token_id 151643 --dtype fp32
uv run optimum-cli export onnx -m prism-ml/Ternary-Bonsai-8B-unpacked model_fp16 --task text-generation-with-past --pad_token_id 151643 --dtype fp16
uv run python convert_fp8.py --source-model-path model_fp16/model.onnx --output-model-path model_fp8/model.onnx
```

`optimum-cli export onnx` accepts `fp32`, `fp16`, and `bf16` for `--dtype`. Since `--dtype fp8` is not supported, fp8 is generated by post-processing a fp16 ONNX export.

### Recommended Configuration

**`model_q2_to_q4 / CPU`** is the best overall choice.

- Load 37.59s / Generation 6.46s — lightweight and practical
- Output quality is natural and stable
- Significantly faster on subsequent runs due to OS file caching
- Achieves generation speed close to `model_fp8 / DirectML` on CPU alone
- Easily obtained by converting from the q2 download (~1.75 GB)

### DirectML Limitations

The three q2-derived models (`model_q2_to_fp8`, `model_q2_to_q4`, `model_q2_to_q8`) all produce garbled output under DirectML. The cause is not weight precision but the `com.microsoft` custom ops inherited from the q2 graph — primarily `GroupQueryAttention` — which DirectML cannot execute correctly. The same models run correctly on CPU. See [ONNX.md](ONNX.md) for a detailed investigation.

For reliable quality under DirectML, use the safetensors-derived `model_fp16` or `model_fp8`.

### Additional Observations

- `model_fp16` and `model_fp8` produce stable output on both CPU and DirectML, but load time is long on HDD.
- `model_q2_to_q8 / CPU` produces correct output but Generation takes 199.62s — impractically slow.
- `model_q2_to_q8` triggers Optimum warnings about legacy ONNX and missing `position_ids`. This is because the graph has a `num_logits_to_keep` input instead of `position_ids`.

## Notes

- Running `run_onnx.py` on Windows automatically downloads ~1.75 GB of q2 packed ONNX from Hugging Face.
- Additional disk space required: ~3.5 GB for q4, ~7 GB for q8, ~7.6 GB for fp8 conversion.
- On Windows with DirectML, do not install `onnxruntime` alongside `onnxruntime-directml`. The standard build takes precedence at load time and hides `DmlExecutionProvider`.
- The `onnxruntime-directml` package varies in DML provider availability by version; this repository pins a verified version.
- On Windows, ONNX Runtime cannot follow symbolic links inside the Hugging Face cache, so [download_q2.py](download_q2.py) copies files as real files under `model_q2/`.
- The `sys_platform == 'win32'` condition in `pyproject.toml` ensures `onnxruntime-directml` is only installed on Windows.
